Task 1.1 — Metrics, Alarms & Filters

CloudWatch Architecture & Data Model

CloudWatch is the primary observability plane. Understanding the hierarchy is fundamental: Namespace → Metric → Dimensions → Statistics → Periods.

Namespace: "AWS/EC2"  |  "MyApp/Production" (custom)
  └── Metric: "CPUUtilization"
        └── Dimensions: {InstanceId: "i-abc", AutoScalingGroupName: "web-asg"}
              └── DataPoints: [(timestamp, value, unit), ...]
                    └── Statistics: Average | Min | Max | Sum | SampleCount
                          └── Period: 60s | 300s (min=1s for high-resolution)

Retention:   High-res custom (1s) → 3h    |  1-min data → 15 days
             5-min data → 63 days          |  1-hour data → 15 months

Standard Monitoring

5-minute granularity, free. EC2 default metrics: CPUUtilization, NetworkIn/Out, DiskReadOps, StatusCheckFailed. Does NOT include memory or disk usage — those require CW Agent.

Detailed Monitoring

1-minute granularity, must be explicitly enabled on EC2. Required for Auto Scaling policies that must react within 1 minute. Costs extra but essential for production fleets.

Composite Alarms — Noise Reduction

Composite alarms combine child alarms with boolean logic. Fire only when multiple conditions are simultaneously true — eliminates alert storms.

# Only alert if BOTH CPU and latency are elevated simultaneously
aws cloudwatch put-composite-alarm \
  --alarm-name "AppDegradation" \
  --alarm-rule "ALARM(\"HighCPU-WebFleet\") AND ALARM(\"HighLatency-ALB\")" \
  --alarm-actions "arn:aws:sns:us-east-1:123456789:PagerDuty" \
  --ok-actions "arn:aws:sns:us-east-1:123456789:SlackOps"
Anomaly Detection: CloudWatch trains an ML model on 2 weeks of metric history and creates an expected-value band. Create anomaly alarms when a static threshold is impractical (e.g., traffic with strong daily/weekly patterns). Alarm fires when metric exits the band.

Logs Insights Query Examples

# Top error types and frequency
fields @timestamp, @message
| filter @message like /ERROR/
| parse @message "ERROR: *" as errorType
| stats count() as freq by errorType
| sort freq desc | limit 20

# Lambda cold start analysis

| stats avg(@initDuration) as avgInit, max(@initDuration) as maxInit,

Task 1.2 — Identify & Remediate Issues

CloudTrail — Auditing & Forensic Investigation

Management Events

Control-plane API calls. Enabled by default. First trail per Region is free. CreateBucket, RunInstances, AttachRolePolicy. Split read/write for cost control.

Data Events

Data-plane operations. Off by default; extra cost. S3 object ops, Lambda invocations, DynamoDB item-level. Essential for data exfiltration investigations.

Insights Events

Detects unusual API call volume vs. baseline. Writes to separate S3 prefix. Example: 100× normal DescribeInstances = recon detection.

🔍

Runbook: Investigate Unauthorized API Activity

1
Scope the incident — define time window and affected resource ARNs. Start with CloudTrail Event History in console (last 90 days, management events).
2
Query CloudTrail Lake or Athena on S3 trail. Pattern: WHERE eventName='AssumeRole' AND errorCode IS NULL AND requestParameters LIKE '%target-role%'
3
Decode caller identity — userIdentity.type: Root, IAMUser, AssumedRole, FederatedUser. For AssumedRole: check userIdentity.sessionContext.sessionIssuer.arn for the origin role.
4
Check errorCode — absence = API call succeeded. AccessDenied = blocked by policy. Correlate resource ARN from requestParameters.
5
Pivot to GuardDuty — check for UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration.OutsideAWS or Recon:IAMUser/MaliciousIPCaller.
6
Contain — revoke active sessions by attaching an explicit deny policy. For compromised EC2: detach IAM role, isolate with restrictive SG, take memory dump if needed.
7
Document & prevent recurrence — rotate credentials, update SCPs, create EventBridge rule to alert on similar activity, file post-mortem.
CloudTrail Lake SQL Query: SELECT eventTime, userIdentity.arn, eventName, errorCode FROM <datastore-ARN> WHERE eventName IN ('AttachRolePolicy','PutRolePolicy','CreateAccessKey') AND eventTime > DATE_ADD('day',-1,NOW()) — detect privilege escalation attempts.

EC2 Placement Groups

Cluster

Single AZ, low-latency, 10Gbps+ between instances. Use for HPC, tightly coupled workloads. Risk: single rack = single failure domain.

Spread

Each instance on a distinct hardware rack. Max 7 per AZ. Use for critical instances needing maximum isolation (e.g., 3-node Zookeeper).

Partition

Groups on separate racks, up to 7 partitions per AZ. Use for distributed systems: Kafka, Cassandra, HDFS. Each partition = separate fault domain.

🎯 Domain 1 Exam Tips & Common Traps
  • Memory/disk metrics require CW Agent — CloudWatch does NOT automatically collect them. This appears in every exam sitting.
  • Composite alarms evaluate alarm states, not metrics directly. You need child metric alarms first.
  • CloudTrail Event History = 90 days, management events only. For data events or full history, you need an S3 trail.
  • Subscription filter limit = 2 per log group. Use Kinesis as a fan-out layer for more consumers.
  • Always configure DLQ on EventBridge targets — without it, failed deliveries are silently dropped.
  • SSM Session Manager requires no inbound SG rules. If asked for secure EC2 access without SSH/RDP, answer is Session Manager.
  • X-Ray default sampling = 5% + 1 req/sec. For payment flows, create custom sampling rule at 100%.
  • gp3 is strictly better than gp2 in most cases — cheaper and allows independent IOPS/throughput provisioning.
  • EBS VolumeQueueLength consistently >1 = I/O bottleneck. Upgrade volume type or instance EBS bandwidth.
Exam Questions — Domain 1
⚡ Scenario · Multiple Choice
A CloudOps engineer must ensure individual CloudWatch alarms trigger SNS AND run SSM Automation to restart EC2. However, during incidents, 8 alarms fire simultaneously, creating an alert storm. What is the MOST operationally correct architecture?
  • A.Configure each alarm with both SNS and SSM Automation as direct actions.
  • B.Create a composite alarm combining the individual alarms with OR logic. Add SNS and SSM actions only to the composite alarm.
  • C.Create an EventBridge rule matching all ALARM state changes and route to both SNS and SSM.
  • D.Use CloudWatch Anomaly Detection on all alarms and act only on anomaly alarms.
Correct: B. Composite alarms enter ALARM state based on a boolean expression across child alarm states — this means a single notification/runbook fires per incident, not N times. Option A causes N separate notifications for one incident. Option C still triggers on every individual alarm transition. The key benefit of composite alarms is noise suppression.
⚡ Scenario · Multiple Choice
A DynamoDB table shows ThrottledRequests consistently above zero during flash sales despite provisioned mode with Auto Scaling targeting 70% utilization. What is the root cause and correct fix?
  • A.Lower Auto Scaling target utilization to 40% to provision capacity sooner.
  • B.Add DynamoDB read replicas in the same Region.
  • C.Auto Scaling has ~1 minute reaction time and cannot handle instantaneous spikes. Switch to on-demand mode for flash sale periods, or pre-warm with scheduled scaling before the event.
  • D.Enable DAX to cache reads and reduce consumed RCU.
Correct: C. DynamoDB Auto Scaling cannot react instantaneously — it has ~1-minute delay. A flash sale spike causes throttling before capacity is added. Solutions: (1) on-demand mode handles any request rate instantly with no capacity planning, or (2) scheduled scaling to pre-provision before the event. DAX (D) reduces read latency but doesn't solve write throttling, and flash sales involve writes too.
⚡ Scenario · Multiple Choice
An application writes structured JSON logs to CloudWatch Logs. The security team needs an alert within 2 minutes when a log entry contains "action": "DELETE" AND "userId": "admin" simultaneously. What is the MOST cost-effective solution?
  • A.Create a CloudWatch Metric Filter with pattern {($.action = "DELETE") && ($.userId = "admin")}, create a CloudWatch Alarm on the metric, configure SNS as the alarm action.
  • B.Subscription filter sending all logs to Lambda, which parses JSON and publishes to SNS.
  • C.Schedule a Logs Insights query every 2 minutes via EventBridge, trigger Lambda to check results.
  • D.Enable CloudTrail data events and create an EventBridge rule for the S3 bucket events.
Correct: A. CloudWatch Metric Filters support JSON field matching with AND/OR syntax natively. This is serverless, near-real-time (<1 min), and free. Option B works but Lambda invocations for every log event are expensive and complex. Option C is polling with minimum 2-min lag guaranteed. Option D is irrelevant — CloudTrail is for API calls, not application JSON logs.

🎯 Domain 2 Exam Tips & Common Traps
  • RDS Multi-AZ standby is NOT readable. Only Aurora Replicas are readable. Multi-AZ standby is passive for HA only.
  • Read Replicas do NOT auto-failover. You must manually promote them. For automatic failover, use Multi-AZ.
  • Aurora failover = ~30s; RDS Multi-AZ = ~60–120s (DNS TTL). RDS Proxy reduces app-visible downtime to seconds by hiding the DNS flip.
  • DynamoDB PITR restores to a NEW table. You must redirect the application connection. In-place restore is NOT possible.
  • EBS snapshot restore has initialization latency. First access to uninitialized blocks reads from S3. Use Fast Snapshot Restore to pre-warm per AZ.
  • Automated RDS backups are deleted when the DB is deleted. Always take a final manual snapshot before deleting an RDS instance.
  • Scheduled Scaling runs BEFORE demand. Predictive Scaling also proactively scales using ML. Both complement target tracking for known cyclical patterns.

  • GuardDuty detects; it does NOT remediate. You must set up EventBridge → Lambda/SSM to act on GuardDuty findings.

  • Permission Boundaries restrict maximum permissions — they do NOT grant permissions. A boundary of S3:* doesn't grant S3 access; it only limits what identity policies can grant.

Runbook: CloudFront Troubleshooting Decision Tree

1
Check X-Cache header: Hit from cloudfront = served from edge. Miss from cloudfront = forwarded to origin. RefreshHit = revalidated (304 Not Modified).
2
Always MISS? Inspect cache key. Unique Authorization headers, User-Agent, or query strings in cache key = 0% hit rate. Each unique combination is a separate cache entry. Simplify the cache policy.
3
Origin sending no-cache? CloudFront respects origin Cache-Control headers. If origin sends no-store or max-age=0, CloudFront won't cache regardless of CloudFront TTL settings. Fix at the origin.
4
5xx errors from origin? Check CloudFront access logs: field x-edge-result-type and x-edge-detailed-result-type. Check origin health (EC2/ELB). Review origin response timeout config.
5
403 Forbidden? Check: (a) S3 Origin Access Control (OAC) — bucket policy must allow CloudFront service principal. (b) WAF Web ACL blocking the request — check WAF sampled requests. (c) Geo restriction.

VPC Flow Logs — Athena Queries

-- All REJECTED traffic in last 24 hours
SELECT srcaddr, dstaddr, dstport, protocol, packets
FROM vpc_flow_logs
WHERE action = 'REJECT'
  AND from_iso8601_timestamp(start) > current_timestamp - interval '24' hour
ORDER BY packets DESC LIMIT 100;

-- Detect port scanning (many unique dst ports from same src)
SELECT srcaddr, COUNT(DISTINCT dstport) as unique_ports, SUM(packets) as packets
FROM vpc_flow_logs
WHERE action = 'REJECT'
  AND from_iso8601_timestamp(start) > current_timestamp - interval '1' hour
GROUP BY srcaddr
HAVING COUNT(DISTINCT dstport) > 20
ORDER BY unique_ports DESC;

🎯 Domain 5 Exam Tips & Common Traps
  • NACLs are stateless — allow ephemeral ports (1024–65535) outbound. Most connectivity issues in NACL-heavy questions come from missing outbound rules.
  • VPC Peering is NOT transitive. A→B and B→C does not allow A→C. Use Transit Gateway for hub-and-spoke or any transitive routing need.
  • S3/DynamoDB Gateway Endpoints are free. Interface Endpoints cost ~$7/month. Always prefer Gateway Endpoints for S3 and DynamoDB from private subnets.
  • NAT Gateway MUST be in a PUBLIC subnet with an Elastic IP. Placing NAT GW in a private subnet is a common architecture mistake that results in no outbound internet connectivity.
  • CloudFront MISS despite correct TTL → check origin Cache-Control headers. CloudFront respects no-store from origin regardless of distribution settings.
  • Global Accelerator = static anycast IPs. Use when clients need to allowlist static IPs, or for non-HTTP protocols (UDP, gaming, IoT).
  • Latency routing ≠ Geolocation routing. Latency routes to lowest-latency AWS Region. Geolocation routes based on the user's geographic location regardless of latency.
  • Flow Logs ACCEPT but traffic still fails = application layer problem. The network path is open. Check the application, DB connection string, or listen port next.

⚡ Scenario · Multiple Choice
EC2 instances in a private subnet make HTTP calls to external APIs via NAT Gateway. The company wants to inspect and block these calls based on domain names (e.g., block *.malware.com) without breaking legitimate traffic. What should the engineer configure?
  • A.Add NACL deny rules for IP addresses of known malicious domains.
  • B.Configure a WAF Web ACL on the NAT Gateway to inspect outbound HTTP traffic.

A CloudFront distribution serves a dynamic API where responses vary per authenticated user (JWT in Authorization header). The cache hit ratio is consistently 0%. What is the root cause and correct fix?
  • A.Dynamic APIs should not use CloudFront. Remove it from the architecture.
  • B.Create a CloudFront invalidation for all paths every 5 minutes.
  • C.The Authorization header is included in the cache key, creating a unique cache entry per user token. Remove Authorization from the cache key. Use an Origin Request Policy to still forward it to origin. For paths that cannot be shared, set TTL=0.

Correct: C. When Authorization header is in the cache key, every unique JWT = unique cache entry = 0% hit rate. For truly per-user content that cannot be shared, set min TTL=0 on that cache behavior. Option D (Origin Shield) consolidates cache misses but cannot help when effective TTL is 0 due to unique cache keys on every request.
