TASK 1.1 — Metrics, Alarms & Filters

CloudWatch Architecture & Core Concepts

CloudWatch is the central observability hub for AWS. Data flows from AWS services, EC2 instances (via the CloudWatch Agent), and custom applications into a hierarchical namespace/metric/dimension structure. Understanding the data model is essential for the exam.

Standard vs. Detailed Monitoring

Standard: 5-minute granularity, free for most services.
Detailed: 1-minute granularity, must be explicitly enabled on EC2. Required for aggressive Auto Scaling policies.

Custom Metrics & High Resolution

Use put-metric-data CLI or SDK. High-resolution custom metrics: 1-second granularity. Standard custom metrics retained for 15 months; high-resolution for 3 hours at 1-second, then aggregated.

CloudWatch Agent Configuration

The unified CloudWatch agent replaces both the legacy CloudWatch Logs agent and the Systems Manager SSM agent for metrics. Install via SSM Run Command or bake into your AMI.

# Install agent via SSM Run Command (Linux)
aws ssm send-command \
  --document-name "AWS-ConfigureAWSPackage" \
  --parameters '{"action":["Install"],"name":["AmazonCloudWatchAgent"]}' \
  --targets "Key=tag:Env,Values=production"

# Agent config: collect custom metrics + logs
{
  "metrics": {
    "namespace": "MyApp/Production",
    "metrics_collected": {
      "mem": { "measurement": ["mem_used_percent"] },
      "disk": { "measurement": ["disk_used_percent"],
               "resources": ["/"] }
    }
  },
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          { "file_path": "/var/log/app/*.log",
            "log_group_name": "/app/production",
            "log_stream_name": "{instance_id}" }
        ]
      }
    }
  }
}
Exam Trap: Memory and disk utilization are NOT automatically collected by CloudWatch for EC2. You must install and configure the CloudWatch Agent to publish these custom metrics.

Composite Alarms

Composite alarms combine multiple alarms using boolean logic (AND, OR, NOT). Use them to reduce alarm noise — only trigger a page when CPU is high AND latency is high simultaneously.

aws cloudwatch put-composite-alarm \
  --alarm-name "AppDegradation" \
  --alarm-rule "ALARM(\"HighCPU\") AND ALARM(\"HighLatency\")" \
  --alarm-actions "arn:aws:sns:us-east-1:123456789:OpsTeam"

Metric Math & Anomaly Detection

Use Metric Math to create derived metrics (e.g., error rate = errors/requests × 100). Anomaly Detection uses ML to model normal metric behavior and alarm on deviations — ideal when a fixed threshold is impractical.

TASK 1.2 — Identify & Remediate Issues

CloudTrail — Auditing & Troubleshooting

CloudTrail records API calls as events. Events are classified as management events (control plane), data events (S3 GetObject, Lambda Invoke), or Insights events (unusual API activity patterns).

Management Events

Enabled by default. Record operations that modify AWS resources: CreateBucket, RunInstances, PutBucketPolicy. Free for first copy per Region.

Data Events

High-volume, must be explicitly enabled. Cover S3 object-level operations and Lambda invocations. Incur extra cost. Essential for security investigations.

🔍

Runbook: Investigate Unauthorized API Call

1
Filter CloudTrail by event name + time window — use Athena on S3 trail bucket or CloudTrail console event history (last 90 days).
2
Identify the caller identity — check userIdentity.arn, userIdentity.sessionContext (for assumed roles), and sourceIPAddress.
3
Check errorCode field — AccessDenied means the call was blocked; no errorCode means it succeeded. Correlate with the resource ARN in requestParameters.
4
Correlate with GuardDuty findings — GuardDuty may have already detected the anomaly (UnauthorizedAccess, InstanceCredentialExfiltration).
5
Remediate — revoke session via IAM, rotate credentials, update SCPs, or isolate the resource depending on severity.

CloudTrail Lake

CloudTrail Lake ingests events into an event data store and allows SQL-based queries directly — no need to export to S3 and use Athena separately. Events are immutable and retained for 7 years.

Patch Manager — Operational Workflow

# 1. Define patch baseline (custom, or use AWS-provided)
# 2. Create maintenance window with schedule (cron)
# 3. Register targets (tag-based: Env=production)
# 4. Register Run Task: AWS-RunPatchBaseline
# 5. Monitor compliance via SSM → Patch Manager → Compliance

# Check compliance via CLI
aws ssm list-compliance-summaries \
  --filters "Key=ComplianceType,Values=Patch,Type=EQUAL"
Key Exam Concept: SSM Patch Manager uses patch baselines to define which patches are approved. AWS provides default baselines (e.g., AWS-AmazonLinux2DefaultPatchBaseline), but you can create custom baselines with specific CVE exceptions. Patch compliance status is reported back to SSM Compliance.

EC2 Placement Groups

Cluster

Single AZ, low latency, 10Gbps network. Use for HPC, tightly coupled workloads. Risk: single point of failure.

Spread

Instances on distinct hardware racks. Max 7 instances per AZ per group. Use for critical instances needing isolation.

Partition

Groups of instances on separate racks, up to 7 partitions per AZ. Use for Hadoop, Cassandra, Kafka distributed systems.

EBS I/O Optimization: Enable EBS-optimized on the EC2 instance for dedicated bandwidth. Monitor VolumeQueueLength — values consistently >1 indicate throughput saturation. Use BurstBalance for gp2/st1/sc1 to monitor burst credits.

S3 Performance Strategies

  • Multipart upload: Required for objects >5GB; recommended for >100MB. Enables parallelism and resume capability.
  • Transfer Acceleration: Routes uploads through CloudFront edge locations using optimized AWS backbone. Use for cross-Region transfers from distant clients.
  • Prefix-based parallelism: S3 scales to 3,500 PUT/5,500 GET requests per second per prefix. Spread keys across multiple prefixes (e.g., hash-based) to multiply throughput.
  • S3 Byte-Range Fetches: Download specific byte ranges in parallel for large object retrieval.
  • S3 Lifecycle Policies: Transition objects through Standard → Standard-IA → Glacier after configurable days. Reduces storage costs without manual management.

DynamoDB Performance

  • On-Demand vs. Provisioned: On-demand auto-scales instantly (no throttling planning); provisioned is cheaper for predictable workloads. Use Auto Scaling with provisioned to handle gradual growth.
  • Hot Partition Problem: If most traffic hits the same partition key, you'll see ProvisionedThroughputExceededException. Solution: use high-cardinality partition keys or add a random suffix (write sharding).
  • DAX: In-memory caching cluster for DynamoDB. Reduces read latency from milliseconds to microseconds. Ideal for read-heavy, eventually consistent workloads.
  • Global Tables: Multi-Region, multi-active replication. Sub-second replication. Use for globally distributed apps with active-active patterns.

Amazon Managed Service for Prometheus & Managed Grafana

For containerized workloads (EKS, ECS), CloudWatch may not be sufficient. AWS provides fully managed Prometheus (compatible with open-source PromQL) and Grafana.

Managed Prometheus (AMP)

Ingest metrics via Prometheus remote_write from EKS/ECS. Create workspaces per team/environment. Query with PromQL. Integrates with Grafana for visualization. No infrastructure to manage.

Managed Grafana (AMG)

Connect to AMP, CloudWatch, Athena, X-Ray as data sources. Use IAM Identity Center for SSO. Built-in alerting that routes to SNS or PagerDuty. Auto-scales to query load.

EXAM QUESTIONS — Domain 1
⚡ Scenario Question · Multiple Choice
A CloudOps engineer notices that CPUUtilization for a fleet of EC2 instances peaks at 95% during business hours but only 20% overnight. The fleet uses t3.medium instances. CloudWatch shows CPUCreditBalance dropping to zero by mid-morning. What is the MOST operationally efficient remediation?
  • A. Enable T3 Unlimited mode to allow burstable CPU beyond baseline indefinitely.
  • B. Switch the fleet to m5.xlarge instances to eliminate CPU credits entirely.
  • C. Enable T3 Unlimited mode and monitor CPUSurplusCreditsCharged to ensure costs remain acceptable; schedule a scale-out during business hours with Auto Scaling.
  • D. Add a CloudWatch alarm on CPUCreditBalance and use an EventBridge rule to send a notification to SNS.
Correct: C. T3 Unlimited allows sustained high CPU by borrowing credits and paying a small per-vCPU surcharge. Combined with Auto Scaling, this handles peak without over-provisioning. Option A is incomplete (no cost monitoring). Option B eliminates the problem but is not cost-efficient — m5 instances are significantly more expensive for workloads that are often at 20%. Option D only notifies; it doesn't fix the problem.
⚡ Scenario Question · Multiple Response
A security team requires that any S3 bucket created without server-side encryption triggers an automated response within 5 minutes. Which TWO services/features are REQUIRED to implement this? (Select TWO)
  • A. AWS Config rule with auto-remediation via SSM Automation
  • B. AWS Macie scheduled scan
  • C. Amazon EventBridge rule triggered on Config compliance change
  • D. CloudTrail data event logging for S3
  • E. GuardDuty S3 protection enabled
Correct: A and C. AWS Config continuously evaluates buckets against an encryption rule. When a non-compliant bucket is detected, Config publishes a compliance change event to EventBridge. EventBridge routes this to an SSM Automation document (or Lambda) that enables encryption. This achieves sub-5-minute remediation. Macie (B) detects sensitive data, not configuration compliance. CloudTrail data events (D) log object access, not bucket configuration changes.

Origin Shield sits between CloudFront edge and origin. Collapses requests, reducing origin load by up to 90%. Improves cache hit ratio. Enable per behavior in distribution settings.

ELB Health Checks

ELB health checks determine whether targets receive traffic. Configure: protocol (HTTP/HTTPS/TCP), path (e.g., /health), threshold (2 healthy, 3 unhealthy), and interval (10–300s). Unhealthy targets are deregistered from the load balancer but NOT terminated — ASG handles termination independently via EC2 health check integration.

Connection Draining (Deregistration Delay): ALB and NLB allow in-flight requests to complete before deregistering a target. Default: 300s. Set to 0 for faster deployments in dev; lower in production for rolling deploys to reduce disruption.

TASK 2.3 — Backup & Restore

Backup & Restore Strategies

AWS Backup — Centralized Backup Management

AWS Backup provides a centralized console to manage backup plans across EC2, EBS, RDS, DynamoDB, EFS, FSx, Storage Gateway, and Aurora. Define backup plans with rules (frequency, retention, vault). Use backup vaults with vault lock (WORM) to prevent backup deletion — required for compliance (HIPAA, PCI-DSS).

RTO vs. RPO — Decision Framework

RPO (Recovery Point Objective) — Maximum acceptable data loss └── How much data can you afford to lose? ├── None → Synchronous replication (Multi-AZ, Aurora Global) ├── Minutes → RDS automated backups + PITR, DynamoDB PITR ├── Hours → Daily EBS snapshots, S3 versioning └── Days → Weekly backups sufficient RTO (Recovery Time Objective) — Maximum acceptable downtime └── How fast must you recover? ├── <1 minute → Multi-AZ failover, Aurora serverless, Route 53 failover ├── <15 minutes → Pilot Light DR pattern ├── <1 hour → Warm Standby DR pattern └── Hours → Backup & Restore DR pattern

EBS Snapshot Strategy

  • Incremental snapshots: Only changed blocks since last snapshot are stored. First snapshot is full; subsequent are incremental. Delete intermediate snapshots safely.
  • Fast Snapshot Restore (FSR): Pre-warms snapshots to eliminate initialization latency when creating a volume. Costs extra; enable per AZ.
  • Data Lifecycle Manager (DLM): Automate snapshot creation and deletion with tag-based policies. Cross-account/cross-region copy supported.

DynamoDB Point-in-Time Recovery (PITR)

PITR provides continuous backups for the last 35 days. Restores to any second within that window into a NEW table (you then redirect traffic). Enable with one click; no impact on table performance. Separate from on-demand backups (which are manual snapshots with no expiry).

S3 Versioning & MFA Delete

Enable versioning + MFA Delete on critical S3 buckets. Versioning protects against accidental deletes/overwrites. MFA Delete requires the root account MFA token to permanently delete versions. Combined with S3 Object Lock (WORM) for regulatory compliance.

EXAM QUESTIONS — Domain 2
⚡ Scenario Question · Multiple Choice
A company has an RDS MySQL Multi-AZ database. The primary instance in us-east-1a fails. The application experiences a connection error for 90 seconds before reconnecting. The team wants to reduce this to under 30 seconds. What should they do?
  • A. Convert the Multi-AZ deployment to a Read Replica for faster failover.
  • B. Enable Enhanced Monitoring to get faster failure detection.
  • C. Place an RDS Proxy in front of the database and configure the application to connect through it.
  • D. Enable Aurora Serverless v2 which provides sub-second failover.
Correct: C. RDS Proxy maintains persistent connection pools to the database and automatically routes connections to the new primary after failover without requiring the application to re-establish connections. This reduces failover impact from the 1-2 minute DNS propagation window to near-zero for active connections. Option A is wrong — Read Replicas don't provide automatic failover. Option D would require migrating to Aurora, which is a bigger change. Option B only improves monitoring, not failover speed.

CloudFormation Troubleshooting Patterns

🛠

Runbook: Debug CREATE_FAILED Stack

1
Open Stack Events — filter for CREATE_FAILED status. The first failure is the root cause; subsequent ROLLBACK events are secondary.
2
Read the Status Reason — for IAM errors look for is not authorized to perform. For subnet errors: Not enough IP addresses. For dependencies: Resource not in a valid state.
3
Set DisableRollback=true on next attempt to preserve resources in failed state for inspection.
4
Check CloudTrail for the API calls made during stack creation — confirms which actions were attempted and which failed.
5
Fix the template, update IAM, expand subnets (CIDR), or resolve dependency ordering with DependsOn.

AWS CDK vs. CloudFormation

CloudFormation

JSON/YAML template. Declarative. Native AWS state management. Best for ops teams; full drift detection; StackSets. The underlying engine for CDK and SAM.

AWS CDK

TypeScript/Python/Java/Go code. Generates CFN templates. L2/L3 constructs provide opinionated defaults. cdk synth → template. cdk deploy → runs CFN. Better for developer-led infrastructure.

AWS RAM — Resource Sharing Across Accounts

RAM allows sharing of VPC subnets, Transit Gateways, Route 53 Resolver rules, Aurora clusters, and more across AWS accounts in the same Organization. Shared subnets are common for centralized networking — consumers launch resources into shared subnets without owning the VPC.

EC2 Image Builder & AMI Pipelines

TASK 4.1 — Security & Compliance Tools

IAM Deep Dive — Policies, Roles & Troubleshooting

IAM Policy Evaluation Logic

When an IAM principal makes an API call, AWS evaluates policies in this order: 1. EXPLICIT DENY anywhere? → DENY (final) 2. SCP (Service Control Policy) allows? → if not → DENY 3. Resource-based policy allows? (for cross-account: also needs identity allow) 4. IAM Permission Boundary allows? 5. Session Policy allows? (for assumed roles) 6. Identity-based policy allows? → ALLOW 7. Otherwise → IMPLICIT DENY Key Rule: Explicit DENY always wins, regardless of any ALLOW. Permission boundaries restrict the maximum permissions — they don't grant.

IAM Policy Troubleshooting Patterns

SymptomInvestigation ToolLikely Cause

Role can't be assumedCloudTrail AssumeRole eventsTrust policy conditions (MFA required, wrong principal ARN, wrong externalId)

EC2 can't call KMSIAM Access Analyzer, KMS key policyEC2 instance profile role missing kms:Decrypt; key policy doesn't include role
# IAM Policy Simulator: test policy effects via CLI
aws iam simulate-principal-policy \
  --policy-source-arn "arn:aws:iam::123456789:role/AppRole" \
  --action-names "s3:GetObject" "kms:Decrypt" \
  --resource-arns "arn:aws:s3:::my-bucket/*"

Permission Boundaries

Permission boundaries are IAM managed policies attached to a role/user that define the maximum permissions it can have. Even if a policy grants s3:*, if the boundary only allows s3:GetObject, only GetObject is effective. Use to safely delegate IAM permissions to developers without granting privilege escalation.

MFA & Federation

MFA Enforcement

Use a condition in SCP or IAM policy: "Condition": {"BoolIfExists": {"aws:MultiFactorAuthPresent": "false"}} with Effect: Deny to block actions without MFA. Except: attach/manage their own MFA device (bootstrap permissions).

IAM Identity Center (SSO)

Centralized SSO for multi-account access. Permission sets define access level. Assign to users/groups per account. Supports external IdPs (Okta, Azure AD). Short-lived credentials via AWS SSO → no long-term access keys needed.

SCPs — Service Control Policies

SCPs apply at the OU or account level in AWS Organizations. They act as a guardrail — they don't grant permissions, they restrict what identity-based policies can grant. The most powerful tools for multi-account governance: prevent disabling CloudTrail, restrict Regions, require encryption tags.

# SCP: Deny all actions outside us-east-1 and us-west-2
{
  "Effect": "Deny",
  "NotAction": [
    "iam:*", "sts:*", "support:*",
    "cloudfront:*", "route53:*"
  ],
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "aws:RequestedRegion": ["us-east-1", "us-west-2"]
    }
  }
}

VPC Endpoints

Gateway Endpoints

S3 and DynamoDB only. Free. Add a route to the route table pointing to the endpoint prefix list. Traffic stays on AWS backbone, doesn't use NAT GW. Most cost-effective for S3/DynamoDB from private subnets.

Interface Endpoints (PrivateLink)

Creates an ENI with private IP in your subnet. Supports 100+ AWS services (SSM, Secrets Manager, CloudWatch, etc.). Costs ~$0.01/hour + data charges. Required when instances can't reach public internet.

Runbook: Troubleshoot CloudFront Cache Miss / Stale Content

1
Check response headers: X-Cache: Hit from cloudfront or Miss from cloudfront. If always MISS, investigate cache policy.
2
Review the Cache Policy — are query strings, headers, or cookies included in the cache key that vary per request? Each unique combination = separate cache entry = low hit rate.
3
Check Origin response headers — if origin sends Cache-Control: no-cache or no-store, CloudFront won't cache regardless of TTL setting.
4
For stale content: create an invalidation (/path/to/file or /*). Propagates to all edge locations within ~60s.
5
For 5xx errors from origin: check custom error pages configuration and origin response timeout settings. Review CloudFront access logs in S3 for error patterns.

Global Accelerator

Global Accelerator provides two static anycast IP addresses that route traffic through the AWS global network backbone to the nearest healthy endpoint. Unlike CloudFront, it is not a cache — it optimizes routing for TCP/UDP applications (gaming, IoT, VoIP, non-HTTP APIs). Use when you need consistent latency and failover for non-HTTP workloads.

Route 53 & DNS Configuration

Route 53 Resolver

Route 53 Resolver handles DNS resolution for hybrid environments. Inbound endpoints allow on-premises DNS servers to resolve AWS Route 53 private hosted zone records. Outbound endpoints allow VPC instances to forward DNS queries for on-premises domains to on-premises resolvers.

# Resolver Rule: Forward corporate.internal to on-prem DNS
{
  "DomainName": "corporate.internal",
  "RuleType": "FORWARD",
  "TargetIps": [
    {"Ip": "10.0.1.53", "Port": 53},
    {"Ip": "10.0.2.53", "Port": 53}
  ],
  "ResolverEndpointId": "rslvr-out-XXXX"
}

Route 53 DNS Firewall

Inspects outbound DNS queries from VPC resources. Block known malicious domains (using AWS-managed or custom domain lists). Alert on suspicious queries. Prevents malware C2 communication and DNS exfiltration attacks. Configure per VPC.

VPC Flow Logs Analysis

# Flow log format key fields:
# version accountId interfaceId srcAddr dstAddr srcPort dstPort
# protocol packets bytes windowStart windowEnd action log-status

# ACCEPT = allowed by SG AND NACL
# REJECT = blocked by SG or NACL

# Athena query: find all REJECTED traffic to port 443
SELECT srcaddr, dstaddr, srcport, dstport, action, packets
FROM vpc_flow_logs
WHERE dstport = 443
  AND action = 'REJECT'
  AND day = '2024/01/15'
ORDER BY packets DESC
LIMIT 50;
Flow Logs Cannot Capture: Traffic to/from 169.254.169.254 (IMDS), traffic to/from Amazon DNS server (169.254.169.253), DHCP traffic, Windows activation traffic, traffic mirrored by Traffic Mirroring.

⚡ Scenario Question · Multiple Response
A company has VPC Flow Logs enabled but cannot determine why EC2 instances in a private subnet are unable to communicate with an RDS instance in the same VPC. The flow logs show ACCEPT entries for the traffic. What are TWO likely causes? (Select TWO)
  • A. The NACL on the private subnet is blocking the traffic.
  • B. The RDS security group does not allow inbound traffic on port 3306 from the EC2 security group.
  • C. VPC Flow Logs are not capturing traffic between resources in the same subnet.
  • D. The application is connecting to the wrong endpoint — using the read replica endpoint instead of the primary.
  • E. The RDS instance is in a different VPC and requires VPC peering.
Correct: B and D. Flow logs show ACCEPT, which means the security groups and NACLs ARE allowing traffic at the network level — so options A and E are incorrect.
